# Commit & release conventions (for humans and AI agents)

The release pipeline in [.github/workflows/build.yml](../.github/workflows/build.yml) cuts a release **only when `package.json` version changes on `main`**. Everything else is just CI. Keep this loop clean and releases happen automatically.

## Commit messages

- **Sentence-case title** (no conventional-commits prefixes like `feat:` — the repo's existing style doesn't use them).
- Focus on the **why**, not the what. Diff shows the what.
- ~1–2 sentences of body if helpful. Skip for trivial changes.
- Group related changes into one commit instead of many small ones — the release notes inherit from commit/PR titles.

Good:
> Fix infinite loop in dialog overlay-stack registration
>
> Effect depended on the whole overlayStack memo which itself rebuilt on every register/unregister. Depend on the stable callables instead.

Bad:
> update dialog.tsx

## When to bump the version

The version field is the only release trigger. Bump it when you have a coherent set of changes you want to ship to users.

| Change class | Bump |
| --- | --- |
| Bug fixes only | `patch` — `0.1.0 → 0.1.1` |
| User-visible features, backward compatible | `minor` — `0.1.0 → 0.2.0` |
| Breaking UX, config, or storage schema changes | `major` — `0.1.0 → 1.0.0` |

Keep it plain semver. Pre-1.0 convention: `minor` for features, `patch` for fixes; breaking changes in minor are tolerable until 1.0.

## Bumping the version

```bash
npm version patch   # or minor / major
```

This edits `package.json`, commits with message `v<version>`, and creates a local tag. Push the commit (the local tag is optional — the workflow publishes its own tag when it promotes the draft release):

```bash
git push
```

Or, if you want the local tag pushed too:

```bash
git push && git push --tags
```

## What the workflow does on each push

```
push to main
├── detect-release: did package.json version change? (no tag v<version> yet)
│     ├── no  → just run CI (typecheck, test, build). No release.
│     └── yes → build with --publish always. Electron-builder uploads
│               artifacts to a draft GitHub Release named v<version>.
└── build: runs on mac / win / linux in parallel.
    └── publish-release: when all three succeed, promote the draft to
          published, regenerate notes from commits since the previous tag.
```

If any one platform fails, the draft is left in place so you can inspect it. Re-push (with the same version) won't help — the draft already exists; delete it first, or bump to a new version.

## Release notes

Auto-generated by `gh release edit --generate-notes` using [.github/release.yml](../.github/release.yml) for categorization.

The config groups PR titles by label. If you want a commit/PR to show up under "Features" vs "Bug fixes", either:

1. When merging a PR, apply a label (`feat`, `fix`, `perf`, `refactor`, `docs`) — or
2. Skip labels and everything lands under "Other".

Title quality directly becomes release note quality. Write PR/commit titles as user-facing release notes.

## Rules for AI agents

- **Don't bump `version`** unless the human explicitly asks for a release. A version bump on an unintended commit cuts a release.
- **Don't rewrite existing release notes or tags.** They're immutable once published.
- **When adding features, group them into one commit** if possible. Multiple trivial commits produce noisy release notes.
- **Write commit titles in the voice of a release note** — "Add X", "Fix Y when Z", not "update files" or "wip".
- **If a change could be a breaking user-visible change** (storage migration, removed shortcut, removed feature flag), call it out explicitly in the commit body so the human releasing it can decide on a major bump.
- **Never commit secrets.** Signing certs and API tokens go in GitHub Actions secrets (see [release-setup.md](release-setup.md)), never in the repo.
- **Never commit `dist/`, `out/`, or `node_modules/`.** Gitignore covers this already — don't unignore it.
- **Don't push directly to main for release-worthy changes.** Open a PR so labels can drive the release-notes categorization.

## Quick reference

```bash
# ship a patch release
npm version patch && git push

# ship a minor release (new feature, backward-compatible)
npm version minor && git push

# ship a major release (breaking change)
npm version major && git push

# watch the workflow
gh run watch

# see the published release
gh release view v$(node -p "require('./package.json').version") --web
```
